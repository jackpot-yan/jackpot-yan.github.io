## 1. Mysql基础
### 1. 语句的执行流程
1. 第一步: 客户端发送sql语句到Mysql服务器
2. 第二步: Mysql服务器的连接器开始处理请求，跟客户端建立连接，获取权限，管理连接
3. 第三步: 解析器开始对sql语句解析，检查是否符合sql语句规则,确保库，表，列都存在，并处理语句中的名称解析和权限验证
4. 第四步: 优化器负责确定sql语句的执行计划(最高效执行),这包括使用索引，以及决定表之间的连接顺序等
5. 第五步: 执行器会调用存储引擎的api来执行读写
6. 第六步：执行结果返回给客户端
7. 客户端接收执行结果

## Mysql日志(重要)
### 1. Mysql日志有哪些
1. 错误日志(error Log):
记录Mysql服务器启动，运行或停止时出现的问题
2. 慢查询日志(slow Query log):
记录执行时间超过long_query_time值的所有语句,这个值可配置，默认情况下，慢查询日志功能是关闭的
3. 一般查询日志(general query log):
记录所有Msyql服务器的连接信息及所有执行语句，不论这些语句是否修改了数据
4. 二进制日志(binary log):
记录了所有修改数据库状态的sql语句，以及每个语句的执行时间，如INSERT,UPDATE，但是不包括SELECT和SHOW这类操作
5. Redo log(InnoDB特有):
记录了对于innoDB表的每个写操作，不是sql级别的，而是物理级别的，主要用于崩溃恢复
6. Undo log(InnoDB特有)
记录数据被修改前的值，用于事务的回滚
### 2. bin log和redo log的区别
1. bin log会记录所有与数据库有关的日志记录，包括InnoDB, MyISAM等存储引擎的日志，而redo log只记录InnoDB引擎的日志
2. 记录的内容不同，bin log记录的是关于一个事务的具体操作内容，即该日志是逻辑日志，而redo log记录的是关于每个页的更改的物理情况
3. 写入时间不同，bin log仅在事务提交前进行提交，也就是只写磁盘一次，而在事务进行的过程中，却不断有redo entry被写入redo log中
4. 写入方式不同，redo log是循环写(类似光盘,会被覆盖)，bin log是追加写(会新开文件)
场景: UPDATE提交(两阶段提交)，保证redo log和bin log一致性:
查找数据->执行器修改数据->写入内存数据页(脏页)->写入redo log(状态为perpare)->写入bin log->redo log状态改为commit->异步刷新入磁盘(check point)
## 索引(重要)
1. 索引为什么能够加快查询
没有索引的话，数据库会全表扫描，这意味着它必须读取表中的每一行数据来查找匹配行(时间效率为On)，有了索引的话，就可以直接跳到索引指示的位置，而不必全表遍历，mysql的InnoDB引擎默认使用B+树做为索引数据结构(时间复杂度为O(logN))
2. 索引的分类
 - 按功能来说,分为
    - 主键索引(表中每行数据的唯一标识，强调列值的唯一性和非空性)
    - 唯一索引(保证每行数据的唯一性，但允许有空值)
    - 普通索引(基本索引类型，加速查询)
    - 全文索引(特定于文本数据的索引)
 - 按数据结构来划分，分为
    - B+树索引,最常见的索引类型，一种将索引值按照一定算法存入一个树形数据结构中(二叉树),每次查询从树的根节点开始，一次遍历叶子节点
    - 哈希索引,基于哈希表的索引，查询效率可以达到O(1),但只适合=和in查询，不适合范围查询
 - 按存储位置划分，分为
    - 聚簇索引,聚簇索引的叶子节点保存了一行记录的所有列信息，也就是说，聚簇索引的叶子节点中，包含了一个完整的记录行
    - 非聚簇索引,它的叶子节点只包含了主键值，通过非聚簇索引首先找到主键，然后通过主键再到聚簇索引中找到记录行，这个过程被称为回表
3. 创建索引注意事项
 - 选择合适的列
    - 经常作为查询条件，排序条件，分组条件
    - 区分度低的字段(唯一性)
    - 避免过多的索引
    - 利用前缀索引和索引列的顺序
4. 索引失效场景
 - 在索引列上使用函数或表达式
 - 使用不等号或not操作
 - 使用LIKE或通配符
 - OR操作符
 - 表过小
 - 联合索引不满足最左匹配原则
5. 选择B+树的原因
 - 更高效的磁盘IO
 - 支持范围查询
 - 叶子节点通过指针连接成链表，范围查询可以遍历
 - 查询性能稳定

## 事务
1. 如何保证四个特性
 - 原子性(undo log)
 - 隔离性
 四种隔离级别(MVCC机制)
 - 持久性(写入磁盘)
 - 一致性(前三个特性保证)
 2. 隔离级别
 分为
  - 读已提交
    - 当前事务只能读取已被其他事务提交的数据，可以避免脏读现象
  - 读未提交
    - 读未提交是最低的事务隔离级别，在这个级别,当前事务可以读取未被其他事务提交的数据，以至于会出现下述三个问题
  - 可重复读
    - 确保在同一事务中多次读取相同记录的结果是一致的，即使其他事务做了修改，也不影响当前事务
  - 串行化
    - 最高事务隔离级别，通过强制事务串行执行来避免并发问题，但可能会导致超市和锁竞争问题
3. 脏读，幻读，不可重复读
 - 脏读，指两个事务a,b，事务a读取到了事务b未提交的数据
 - 不可重复读，指在一个事务范围内，两个相同的查询，读取同一条记录，却返回了不同的数据
 - 幻读，事务a查询一个范围的结果集，另一个并发事务b往这个范围中插入/删除了数据，并静悄悄提交，然后事务a再次查询，，两次返回的结果不一致
 4. 如何实现
  - 读已提交和可重复读是通过MVCC机制中的视图(read view)来实现
    - 对于读已提交，每次读取数据前会生成视图，保证每次读操作都是最新数据。
    - 对于可重复读，只在第一次读取时生成视图，后续读操作使用相同视图，保证事务读取的一致性
    - 串行化，事务在读操作时必须先加表级别共享锁，直到事务结束才释放，事务在写操作时，必须先加表级别排他锁，知道事务结束释放
5. MVCC(及其重要，多版本并发控制)
 - 在支持MVCC的数据库当中，当多个用户同时访问数据时，每个用户都可以看到一个在某一时间点之间的数据库快照，并且能够无阻塞的执行查询和修改操作，而不会相互干扰
通过版本链和视图实现
 - 版本链实现，在InnoDB引擎中，每一行数据都有两个隐藏的列:一个是DB_TRX_ID，另一个是DB_ROLL_PTR
    - DB_TRX_ID是保存创建这个版本的事务ID
    - DB_ROLL_PTR是指向undo log日志记录的指针，这个记录包含了改行的前一个版本的信息，通过这个指针可以访问到该行数据的历史版本
